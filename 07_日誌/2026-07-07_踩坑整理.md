# 2026-07-07 踩坑整理

## 定位

本文件整理 `project-004-工程分析` 目前在工程判決資料蒐集、特徵提取、模型分析與回測設計中遇到的問題。狀態分為已大致解決、部分解決、待釐清與建議補強。

## 目前踩過的坑

### 1. 無法批次下載判決書

- 狀態：大部分解決。
- 原問題：一開始無法穩定批次取得判決書，若只能手動下載，樣本量會嚴重不足，也會拖慢後續標註與模型測試。
- 目前進度：已改用爬蟲直接下載 JSON 判決全文，完整資料庫目前包含 `2021-2026判決總表.csv` 與 5,906 件 JSON 判決檔。
- 後續注意：下載流程應保留來源欄位、案號、裁判日期、法院與原始 JSON 路徑，避免後續無法追溯。

### 2. 統計誤差與人工標註成本

- 狀態：部分解決。
- 原問題：若要讓統計誤差足夠低，精標樣本可能需要上百份以上；但逐件閱讀判決、抓金額、抓理由非常花時間。
- 目前進度：已考慮先用判決摘要或 AI 輔助摘要，再從摘要提取特徵。
- 主要風險：摘要可能漏掉關鍵金額、訴訟階段、法院准駁理由或民法第 252 條酌減判斷；若只從摘要提特徵，容易把模型建立在二手資料上。
- 建議作法：採「大樣本粗篩、小樣本精標」：先用 5,906 件全文篩出候選池，再抽 60-120 件精標；AI 摘要只能輔助，最後仍回到全文人工核對。

### 3. 特徵一開始只有布林值，難以支撐複迴歸

- 狀態：部分解決。
- 原問題：最初提取出的特徵多為 `issue_*` 布林值，但希望用複迴歸分析甚至預測判決結果，因此還需要數值欄位或類別欄位。
- 目前進度：已順利提取出部分字串與金額候選特徵。
- 新問題：同一份判決中同一類金額會多次出現，例如契約總價、估價單金額、追加工程款、扣款、保留款、違約金、利息、訴訟費用、不同年份估價或不同請求項目。
- 建議作法：不要只抽「金額數字」，而要標註金額角色，例如 `contract_price`、`claimed_penalty`、`allowed_penalty`、`paid_amount`、`additional_work_amount`、`retention_amount`。無法確定角色的金額只能放在候選欄位，不能直接進模型。

### 4. 模型型態與預測目標不一致

- 狀態：待釐清。
- 原問題：Codex 曾判斷目前提取出的數值不適合直接做複迴歸，建議用邏輯迴歸；但邏輯迴歸目標通常是布林值，和原本希望預估金額或免除違約金天數的方向不同。
- 目前進度：曾嘗試直接指定帶虛變數的複迴歸模型，但執行中因電腦沒電中斷。
- 核心釐清：要先分清楚模型目標。
  - 若目標是「是否酌減」或「是否涉及違約金」，適合用邏輯迴歸、Linear SVM、LightGBM 分類。
  - 若目標是「酌減比例」或「法院准許金額」，才適合用線性迴歸、Ridge、Lasso、Random Forest Regressor 等迴歸模型。
  - 若目標是「判決結果」，必須先定義結果，例如原告勝訴、部分勝訴、法院是否准許違約金、是否酌減，而不能籠統說預測判決。
- 建議作法：主模型先做分類模型，預測 `is_reduced`；進階模型再做 `remaining_ratio = allowed_penalty / claimed_penalty` 的迴歸回測。

## 補充漏點

### 5. 26 筆不是樣本上限，而是起始標註或測試集

- 狀態：已釐清。
- 問題：先前容易把 26 筆爭點標註資料誤解為整個專題的樣本量，導致模型與統計討論過度保守。
- 實際情況：完整資料庫有 5,906 件 JSON 判決，26 筆只是目前 `penalty_cases.csv` 的起始標註骨架。
- 建議作法：報告中應寫明資料層次：5,906 件為母體，關鍵詞篩選後形成候選池，人工精標樣本另行定義。

### 6. 關鍵詞篩選召回高，但精準度需要驗證

- 狀態：待補強。
- 問題：例如 `違約金`、`逾期`、`工期`、`酌減` 等關鍵詞能快速篩出大量候選案件，但命中不一定代表該案真正以逾期違約金酌減為核心爭點。
- 目前粗篩結果：`違約金` 約 1,961 件，`逾期違約金/違約金+逾期` 約 1,606 件，`違約金+酌減/民法第252條` 約 704 件。
- 建議作法：抽樣人工驗證關鍵詞篩選 precision，並建立排除規則，例如排除單純引用、非工程逾期、利息遲延、程序性敘述。

### 7. 上訴審、同案不同審級可能造成資料洩漏

- 狀態：待補強。
- 問題：同一工程爭議可能出現在地院、高院、最高法院，若隨機切分訓練集與測試集，同一案件的不同審級可能同時出現在訓練與測試中，使模型看似表現很好。
- 建議作法：回測時盡量以時間切分或案件群組切分；如果能辨識同案關係，應將同案放在同一切分區。

### 8. 金額欄位容易混入利息、訴訟費、比例與日期

- 狀態：待補強。
- 問題：判決中常有年息百分之五、起息日、訴訟費用比例、假執行擔保金、已付款、扣款等數字，不能直接視為工程金額。
- 建議作法：金額抽取需要搭配上下文窗口與人工角色判定；自動抽取結果只作為候選清單。

### 9. 人工智慧摘要可能降低成本，也可能放大錯誤

- 狀態：待驗證。
- 問題：AI 摘要能加速閱讀，但可能漏掉金額、錯置當事人主張與法院認定，或把上訴意旨當成法院見解。
- 建議作法：摘要流程應要求輸出來源段落、金額依據與不確定標記；人工標註時回查原文，不直接把摘要當作事實。

### 10. 模型回測需要設定基準線

- 狀態：待補強。
- 問題：若只報告模型分數，無法知道模型是否真的優於簡單規則。
- 建議作法：至少建立三個 baseline：
  - 多數類別基準：永遠預測最常見結果。
  - 關鍵詞規則基準：用 `民法第252條`、`酌減` 等詞判斷。
  - 簡單線性基準：Logistic Regression 或 Ridge 作為可解釋模型。

### 11. 電腦中斷與長時間模型搜尋需要檢查點

- 狀態：已發生，待補強。
- 問題：模型跑到一半因電腦沒電關機，會造成時間損失，也可能遺失中間結果。
- 建議作法：長時間訓練或 TPOT/AutoML 搜尋需設定輸出目錄、定期保存結果、限制搜尋時間，並優先用可快速重跑的 baseline 模型。

### 12. 法律預測的表述要保守

- 狀態：持續注意。
- 問題：若報告寫成「預測法院判決」，容易被質疑過度宣稱，且法律判斷涉及事實、證據、程序與法官心證。
- 建議作法：表述為「輔助篩選」、「風險辨識」、「爭點分類」、「酌減可能性回測」或「探索性模型」，避免宣稱可替代法律判斷。

## 建議下一步

1. 明確定義第一個模型目標：建議先做 `是否涉及逾期違約金酌減` 或 `is_reduced`。
2. 建立候選池：從 5,906 件中篩出 `違約金 + 逾期 + 酌減/民法第252條` 案件。
3. 抽樣精標 60-120 件：補 `contract_price`、`delay_days`、`claimed_penalty`、`allowed_penalty`、`key_reason`。
4. 先跑 baseline：關鍵詞規則、Logistic Regression、Ridge/Linear Regression。
5. 再跑進階模型：LightGBM、Random Forest、TPOT 或其他 AutoML，但僅作為輔助比較。
6. 回測採時間切分：例如 2021-2024 訓練、2025 驗證、2026 測試。
7. 報告中明確區分：資料庫母體、候選池、人工精標樣本、模型回測樣本。
